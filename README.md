# Web Cache Deception Demo Project

This project is a four-person group implementation and presentation of the paper **“Web Cache Deception Escalates!”**. The goal is not to attack real websites. The goal is to build a controlled local demo that shows how Web Cache Deception (WCD) can happen when an origin server and a reverse proxy cache disagree about whether a response should be cached.

The demo should run only in a local or explicitly authorized lab environment.

---

## 1. Project idea

In a normal request, the user visits a dynamic page such as:

```text
/profile
```

That page may contain sensitive dynamic content, such as a CSRF token.

In a WCD-style request, the attacker creates a URL that still reaches the same dynamic page on the origin server, but looks like a static file to the cache:

```text
/profile/random_name.css
```

If the origin server returns the dynamic `/profile` page, but the reverse proxy cache treats the URL as a cacheable `.css` file, the sensitive dynamic response may be stored in the cache. A later unauthenticated request to the same attack URL may then receive the cached victim response.

Member 1 has implemented:

### Member 2 — Core Detection Engine

**Main responsibility:** Implement the main DE detection algorithm from the paper.

Member 2 should implement:

- Step 1: Request the original URL twice and compare responses to check whether the page is dynamic.
- Step 2: Use attack URLs generated by Member 3 and check whether the modified URL still returns dynamic content.
- Step 3: Check whether the first attack request is a cache `MISS`, while the repeated request to the same attack URL becomes a cache `HIT`.
- Byte-by-byte or hash-based body comparison between responses.
- Fuzzy cache-header matching using `hit`, `miss`, `cached`, or `caching` keywords.

### Member 3 — Payloads & False Positives

**Main responsibility:** Generate WCD attack payloads and filter false positives.

This is my part.

I implemented the file:

```text
member3_wcd_completed.py
```

My module provides:

- randomized attack filename generation,
- path-confusion payload generation,
- multiple encoding-based payload modes,
- fuzzy cache HIT/MISS header detection,
- Algorithm 2 style false-positive checking,
- optional CSV/JSON export of generated payloads.

The payload generator produces attack URLs such as:

```text
/profile/random.css
/profile%3Brandom.css
/profile%3Frandom.css
/profile%0Arandom.css
/profile%23random.css
/profile%2Frandom.css
/profile%25%33%42random.css
```

The random filenames are used so that repeated tests do not accidentally reuse the same cached poisoned URL.

#### Main functions provided by my part

```python
generate_attack_url(url, mode, extension=".css")
```

Generates one attack URL for one selected path-confusion mode.

```python
generate_attack_urls(url, extensions=(".css",), count_per_mode=1)
```

Generates all attack URLs for a target URL.

```python
infer_cache_status(headers)
```

Returns one of:

```text
HIT
MISS
AMBIGUOUS
UNKNOWN
```

This is useful for Member 2 when checking response headers.

```python
check_false_positive(original_url)
```

Implements the false-positive check. It requests the original URL twice without any attack payload. If the second normal request is already a cache `HIT`, then the page is cacheable even without WCD, so the suspicious finding should be treated as a false positive.

```python
filter_false_positives(original_urls)
```

Convenience function for checking multiple original URLs after Member 2 reports suspicious findings.

## 3. How to run Member 3’s module

Install the dependency:

```bash
pip install requests
```

Generate payloads and run the false-positive check:

```bash
python member3_wcd_completed.py --url http://localhost:8080/profile
```

Only generate payloads, without sending the false-positive test requests:

```bash
python member3_wcd_completed.py --url http://localhost:8080/profile --skip-fp-check
```

Generate payloads and save them as CSV:

```bash
python member3_wcd_completed.py \
  --url http://localhost:8080/profile \
  --output-csv payloads.csv
```

Generate payloads and save them as JSON:

```bash
python member3_wcd_completed.py \
  --url http://localhost:8080/profile \
  --output-json payloads.json
```

## 4. How Member 2 should connect to Member 3

Member 2 can use my module in two places.

### A. During attack URL generation

Instead of writing their own payloads, Member 2 should call:

```python
from member3_wcd_completed import generate_attack_urls

target_url = "http://localhost:8080/profile"
payloads = generate_attack_urls(target_url)

for payload in payloads:
    attack_url = payload.attack_url
    mode = payload.mode
    print(mode, attack_url)
```

Member 2 then sends requests to these attack URLs during Step 2 and Step 3 of the detection algorithm.

### B. During cache header analysis

Member 2 can use:

```python
from member3_wcd_completed import infer_cache_status

status = infer_cache_status(response.headers)
```

This helps classify a response as `HIT`, `MISS`, `AMBIGUOUS`, or `UNKNOWN`.

### C. After suspicious findings are found

After Member 2 finds suspicious WCD cases, they should call my false-positive filter:

```python
from member3_wcd_completed import check_false_positive

result = check_false_positive("http://localhost:8080/profile")

if result.is_false_positive:
    print("Remove this finding:", result.conclusion)
else:
    print("Keep this finding:", result.conclusion)
```

For multiple URLs:

```python
from member3_wcd_completed import filter_false_positives

original_urls = [
    "http://localhost:8080/profile",
    "http://localhost:8080/settings",
]

results = filter_false_positives(original_urls)

for result in results:
    print(result.original_url, result.is_false_positive, result.conclusion)
```

## 6. How Member 4 should connect to Member 3

Member 4 is responsible for the live demo and presentation flow.

Member 4 can use one generated attack URL from my module as the malicious demo link.

Example:

```bash
python member3_wcd_completed.py --url http://localhost:8080/profile --skip-fp-check
```

Then choose one output URL like:

```text
http://localhost:8080/profile/random_name.css
```

Demo flow:

1. Victim browser opens the generated attack URL.
2. The origin server returns the victim’s dynamic `/profile` page.
3. The reverse proxy cache stores the response because the URL looks like a `.css` file.
4. Attacker browser opens the same attack URL without authentication.
5. The attacker receives the cached response and can see the dummy CSRF token.

Member 4 should use only dummy data and a local demo account. The goal is to visually explain the vulnerability, not to attack a real system.
